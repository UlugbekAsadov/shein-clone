## Getting Started

```bash
pnpm install
pnpm dev          # http://localhost:3000
pnpm build        # production build
pnpm lint         # eslint
npx tsc --noEmit  # type-check
```

Before touching code:

- Skim this whole document.
- Skim [AGENTS.md](../AGENTS.md) — that is the canonical rule list.
- Look at a few existing components in `src/components/common/footer/` to see how the rules apply in practice.

## Tech Stack

- **Framework**: Next.js 16 (App Router)
- **Language**: TypeScript (strict)
- **Styling**: Tailwind CSS v4 + `tw-animate-css`
- **UI primitives**: shadcn/ui (Radix under the hood)
- **Icons**: `lucide-react`
- **State / data**: TanStack Query
- **Package manager**: pnpm
- **i18n**: locale-prefixed routes (`/uz`, `/ru`, `/en`)

### What folder does what

| Folder                      | Holds                                    | Notes                                                     |
| --------------------------- | ---------------------------------------- | --------------------------------------------------------- |
| `app/[lang]/_components/`   | Page-scoped components                   | Only used by routes inside `[lang]/`.                     |
| `app/[lang]/_lib/`          | Page-scoped constants, mocks, interfaces | Mirrors `src/lib/` but local to the route.                |
| `components/ui/`            | shadcn/ui primitives                     | Generated by the shadcn CLI. Don't write components here. |
| `components/common/<name>/` | Shared, cross-page components            | One folder per component family.                          |
| `lib/interfaces/`           | All shared TypeScript interfaces         | I-prefixed only.                                          |
| `lib/mock-data/`            | All shared mock data                     | No interfaces here.                                       |
| `lib/constants/`            | All shared constants                     | No mocks here.                                            |
| `lib/helpers/`              | Shared helper functions                  | Create when you have the first one.                       |

## The 11 Rules

The rules below are numbered the same way as in [AGENTS.md](../AGENTS.md). Examples show the **wrong** way (do not write) and the **right** way (do write).

### Rule 1 — Prefer shadcn/ui + lucide-react

Before writing anything custom, check whether shadcn already has it: `Button`, `Dialog`, `Select`, `Input`, `Avatar`, `Badge`, `DropdownMenu`, `ScrollArea`, `Separator`. For icons, use `lucide-react`.

**Wrong**

```tsx
<button className="rounded bg-foreground px-4 py-2 text-background">
  Buy now
</button>
```

**Right**

```tsx
import { Button } from "@/components/ui/button";
import { ShoppingCart } from "lucide-react";

<Button>
  <ShoppingCart className="size-4" />
  Buy now
</Button>;
```

If shadcn doesn't have what you need, ask in the team chat before rolling your own.

---

### Rule 2 — One component per `.tsx` file

A `.tsx` file exports exactly one React component.

**Wrong** (`footer.tsx`)

```tsx
export function Footer() {
  /* ... */
}
function FeatureItem() {
  /* ... */
}
function FooterColumn() {
  /* ... */
}
```

**Right** — split into siblings:

```
footer/
├── footer.tsx          // export function Footer
├── feature-item.tsx    // export function FeatureItem
└── footer-column.tsx   // export function FooterColumn
```

---

### Rule 3 — No comments in code

No `//`, no `/* */`, no JSDoc. If you feel the urge to comment, rename the variable, extract a helper, or split the function.

**Wrong**

```tsx
// Calculate sold count from reviews
const soldCount = product.reviews * 5 + 123;
```

**Right** — let the name carry the meaning:

```tsx
const soldCount = product.reviews * 5 + 123;
```

If a piece of code is genuinely surprising, write a clearer abstraction — don't paper over it with a comment.

---

### Rule 4 — No big components

If a component is getting long (rule of thumb: > 150 lines, or two distinct "areas"), split it. The product preview dialog is a good reference: the parent owns state, every visual section is its own subcomponent.

See [product-preview/](../src/components/common/product/product-preview/) — `product-preview-dialog.tsx` orchestrates, and gallery / color / size / qty / rating live in their own files.

---

### Rule 5 — Group connected components into one folder

When components only make sense together, give them a folder.

**Wrong** — flat siblings in a shared bucket:

```
_components/
├── brand-story-viewer.tsx
├── story-card.tsx
├── progress-bar.tsx
├── hero-carousel.tsx
└── ...
```

**Right** — story components grouped:

```
_components/
├── brand-story-viewer/
│   ├── brand-story-viewer.tsx
│   ├── story-card.tsx
│   └── progress-bar.tsx
├── hero-carousel.tsx
└── ...
```

The folder name is the parent component's name in kebab-case.

---

### Rule 6 — `_lib/` for component-scoped non-component files

Types, constants, mocks, and helpers that only belong to a route or component family live in a sibling `_lib/` folder, not in the `.tsx` file itself.

**Wrong** (`brand-story-viewer.tsx`)

```tsx
const STRIP_DURATION_MS = 10_000;
const TRANSITION_MS = 500;
const SLOT_WIDTH = 380;

export function StoryViewer({ ... }) { ... }
```

**Right**

```
app/[lang]/
├── _components/
│   └── brand-story-viewer/
│       └── brand-story-viewer.tsx     // imports from _lib
└── _lib/
    └── constants/
        └── brand-story.constants.ts   // STRIP_DURATION_MS, etc.
```

---

### Rule 7 — Files inside `_lib/` include their kind in the filename

| Kind                     | Suffix          | Example                    |
| ------------------------ | --------------- | -------------------------- |
| Interface                | `.interface.ts` | `brand.interface.ts`       |
| Constants                | `.constants.ts` | `brand-story.constants.ts` |
| Mock data                | `.mocks.ts`     | `product.mocks.ts`         |
| Helpers                  | `.helpers.ts`   | `price.helpers.ts`         |
| Types (only if you must) | `.types.ts`     | —                          |

The kind suffix makes greps and imports unambiguous.

---

### Rule 8 — Shared non-components live in `src/lib/`, split by kind

`src/lib/` is the shared bucket. **It is split into sibling folders by kind, never mixed.**

```
src/lib/
├── interfaces/       // only *.interface.ts
├── mock-data/        // only *.mocks.ts
├── constants/        // only *.constants.ts
└── helpers/          // only *.helpers.ts
```

**Wrong** — interface and mock in the same folder:

```
src/lib/mock-data/
├── brand.interface.ts   // wrong place
└── brand.mocks.ts
```

**Right**

```
src/lib/
├── interfaces/
│   └── brand.interface.ts
└── mock-data/
    └── brand.mocks.ts
```

---

### Rule 9 — Use `interface IProps` for component prop types

**Wrong**

```tsx
type Props = {
  product: IProduct;
};

export function ProductCard({ product }: Props) { ... }
```

**Right**

```tsx
interface IProps {
  product: IProduct;
}

export function ProductCard({ product }: IProps) { ... }
```

---

### Rule 10 — All interfaces start with `I`

`IProps`, `IProduct`, `IBrand`, `ICategory`, `IShop`, `IDictionary`, etc.

**Wrong**

```ts
export interface Brand {
  id: string;
  name: string;
}
```

**Right**

```ts
export interface IBrand {
  id: string;
  name: string;
}
```

---

### Rule 11 — No `type` aliases

Use `interface` for everything.

For unions or derived types, inline them at the usage site instead of creating a `type` alias.

**Wrong**

```ts
export type Tone = "rose" | "emerald" | "blue" | "amber";

interface IProps {
  tone: Tone;
}
```

**Right** — inline at the prop site:

```ts
export const filterChipTones = {
  rose: "...",
  emerald: "...",
  blue: "...",
  amber: "...",
} as const;

interface IProps {
  tone: keyof typeof filterChipTones;
}
```

**Wrong** — type alias for a derived shape:

```ts
export type Dictionary = Awaited<ReturnType<typeof dictionaries.en>>;
```

**Right** — interface extends the derived type:

```ts
export interface IDictionary extends Awaited<
  ReturnType<typeof dictionaries.en>
> {}
```

**Wrong** — type alias for a locale union:

```ts
export type Locale = (typeof locales)[number];
function hasLocale(v: string): v is Locale { ... }
```

**Right** — inline the derived expression:

```ts
function hasLocale(v: string): v is (typeof locales)[number] { ... }
```

## How-To Guides

### Add a page-scoped component

Used by exactly one route, lives under `app/[lang]/_components/`.

1. Create `app/[lang]/_components/my-feature.tsx`.
2. Inside, declare `interface IProps { … }` and `export function MyFeature(props: IProps) { … }`.
3. Import where it's used: `import { MyFeature } from "./_components/my-feature";`.

If `MyFeature` grows subcomponents (rule 5), promote it to a folder:

```
_components/my-feature/
├── my-feature.tsx
├── my-feature-row.tsx
└── my-feature-toolbar.tsx
```

…and update the import to `./_components/my-feature/my-feature`.

### Add a shared component

Used in more than one place, lives under `components/common/<family>/`.

1. Pick the family (`header`, `footer`, `product`, etc.) or create a new family folder.
2. Add `components/common/<family>/my-thing.tsx` with `interface IProps` + named export.
3. Import: `import { MyThing } from "@/components/common/<family>/my-thing";`.

If multiple closely related components ship together (parent + subcomponents), create a subfolder:

```
common/product/product-preview/
├── product-preview-dialog.tsx
├── product-preview-gallery.tsx
└── ...
```

### Add a new interface

All shared interfaces go in `src/lib/interfaces/`.

1. `src/lib/interfaces/order.interface.ts`:

   ```ts
   export interface IOrder {
     id: string;
     total: number;
     items: IOrderItem[];
   }

   export interface IOrderItem {
     productId: string;
     qty: number;
   }
   ```

2. Re-export from the barrel: add `export * from "./order.interface";` to `src/lib/interfaces/index.ts`.
3. Import via explicit path: `import type { IOrder } from "@/lib/interfaces/order.interface";`.

Component-scoped interfaces (used by exactly one component family) can live in a local `_lib/interfaces/` next to that component instead.

### Add new mock data

All shared mocks go in `src/lib/mock-data/`.

1. `src/lib/mock-data/order.mocks.ts`:

   ```ts
   import type { IOrder } from "../interfaces/order.interface";

   export const sampleOrders: IOrder[] = [
     /* ... */
   ];
   ```

2. Re-export from `src/lib/mock-data/index.ts`: `export * from "./order.mocks";`.
3. Import: `import { sampleOrders } from "@/lib/mock-data";`.

### Add new constants

All shared constants go in `src/lib/constants/`.

1. `src/lib/constants/shipping.constants.ts`:
   ```ts
   export const FREE_SHIPPING_THRESHOLD = 50;
   export const SHIPPING_METHODS = ["standard", "express"] as const;
   ```
2. Import: `import { FREE_SHIPPING_THRESHOLD } from "@/lib/constants/shipping.constants";`.

Page-scoped constants go in `app/[lang]/_lib/constants/` instead.

### Where do I put X?

| Question                                              | Answer                                          |
| ----------------------------------------------------- | ----------------------------------------------- |
| It's a React component used in one route              | `app/[lang]/_components/`                       |
| It's a React component reused across routes           | `components/common/<family>/`                   |
| It's a constant used in one route                     | `app/[lang]/_lib/constants/`                    |
| It's a constant used in many places                   | `src/lib/constants/`                            |
| It's a TypeScript interface for shared data           | `src/lib/interfaces/`                           |
| It's a TypeScript interface for one component's props | inside that component file (`interface IProps`) |
| It's mock data                                        | `src/lib/mock-data/`                            |
| It's a small pure helper used many places             | `src/lib/helpers/` (create folder if missing)   |

## Naming Cheatsheet

| Thing            | Convention                                                     | Example                                      |
| ---------------- | -------------------------------------------------------------- | -------------------------------------------- |
| File (component) | `kebab-case.tsx`                                               | `product-card.tsx`                           |
| File (lib)       | `<name>.<kind>.ts`                                             | `product.interface.ts`, `brand.mocks.ts`     |
| Folder           | `kebab-case/`                                                  | `product-preview/`                           |
| Component        | `PascalCase`                                                   | `ProductCard`                                |
| Interface        | `IPascalCase`                                                  | `IProduct`, `IProps`                         |
| Constant         | `SCREAMING_SNAKE` for primitives; `camelCase` for object/array | `FREE_SHIPPING_THRESHOLD`, `filterChipTones` |
| Hook             | `camelCase` starting with `use`                                | `usePreviewState`                            |
| Tailwind classes | as-is                                                          | `text-sm font-semibold`                      |

## FAQ

**Q: Can I name a prop interface something other than `IProps`?**
A: No. Use `IProps` inside the component file. If you genuinely need to export it (rare), name the exported version after the component, e.g. `IProductCardProps`.

**Q: What about types that can't be expressed as interfaces (unions, conditional types)?**
A: Inline them at the usage site. If a union shows up in three or more places, define a `const` map (see `filterChipTones`) and derive `keyof typeof …` inline.

**Q: I added a tiny helper. Where does it go?**
A: If it's used in one component, put it in that component's file (or its `_lib/helpers/<name>.helpers.ts` if reused inside the family). If it's shared, create `src/lib/helpers/<name>.helpers.ts`.

**Q: Can I leave `// TODO:` comments?**
A: No. Track TODOs in the issue tracker or PR description, not in code.

**Q: I think a rule is wrong for my case. What do I do?**
A: Open an issue (or raise it in the PR). Don't quietly bend the rule. Consistency beats local optimums.

**Q: Where do I edit shadcn primitives?**
A: Usually you don't. If you need to change behavior, wrap the primitive in `components/common/` instead of editing `components/ui/`.

**Q: Why no `type` aliases?**
A: Consistency. With one rule ("interfaces only"), everyone knows where to look. `interface` extends, declaration-merges, and reads predictably across the codebase.

**Q: Where are the rules enforced?**
A: ESLint catches some, PR review catches the rest. If you see a violation in `main`, fix it in a small follow-up PR.
